# AGENTS.md 作成

## 目的

AGENTS.md は README や設計書ではなく、AI コーディングエージェントがそのリポジトリで安全に作業するための作業契約として書く。長い背景説明ではなく、毎回守るべき具体的なルール、実在するコマンド、リポジトリ固有の注意点を残す。

AGENTS.md は多くの環境で毎回のセッションに読み込まれる前提で扱う。万能な知識置き場にせず、ほぼすべての作業に効く最小限の指示と、詳細情報への参照だけを書く。

## ワークフロー

1. 既存の指示を確認する。
   - ルートと作業対象配下の `AGENTS.md` を探す。
   - README、既存ドキュメント、プロジェクトメモ、CI 設定、主要な設定ファイルを読む。
   - 使われている言語、パッケージ管理、ビルド、テスト、lint、format、デプロイの手段をリポジトリから確認する。
   - 外部ドキュメントやチケット URL が渡された場合は、利用できる正規の手段で読む。読み方を推測できない場合はユーザーに確認する。
   - 秘密情報を含みうるファイルは、値を表示せずキー名、存在、用途だけ確認する。

2. 情報を分類する。
   - AGENTS.md に入れるもの: 回答言語、作業方針、実行コマンド、検証方法、技術スタック、ディレクトリ責務、環境変数の扱い、Git 運用、デプロイや外部サービス固有の注意点。
   - AGENTS.md に入れないもの: 長い背景説明、仕様書の全文、古くなりやすい TODO、README と同じ一般説明、未確認のコマンド、抽象的な精神論、細かいコードスタイル規則の羅列。
   - ロードマップやアイデアは、エージェントの作業判断に直接効くものだけ残す。詳しい内容は `docs/roadmap.md` など別文書向きとして扱う。
   - 詳細なビルド手順、テスト方針、設計、スキーマ、ドメイン知識は、必要に応じて別文書へ分け、AGENTS.md から短く参照する。

3. AGENTS.md を書く。
   - 日本語で、短く、命令形で書く。
   - 実在するコマンドだけを書く。
   - ファイルパスや設定名は具体的に書く。
   - 既存のユーザー指示を勝手に削除しない。矛盾があれば新しい指示を優先し、必要ならユーザーに確認する。
   - README の代替にしない。人間向け説明ではなく、エージェントの作業ルールに寄せる。
   - 可能なら 60 行程度を目標にし、長くても 300 行未満に抑える。長くなる場合は詳細文書へ分割する。
   - lint や format で機械的に検証できる内容を自然言語ルールとして増やさない。コマンドや hook への参照にする。

4. 確認する。
   - 最終版を読み返し、曖昧な文や実行不能な指示を削る。
   - 可能なら設定ファイルやタスク定義で記載コマンドが存在することを確認する。
   - すべての行について「毎回の作業で読む価値があるか」を確認し、限定的な内容は別文書への参照に変える。
   - 完了報告では、使った情報源、主な変更点、未確認の前提を短く伝える。

## 推奨構成

詳細なセクション例は `references/agents-md-structure.md` を必要なときだけ読む。通常は次の順序を基本にする。

- 基本方針
- プロジェクト概要
- 技術スタック
- 開発コマンド
- 検証コマンド
- ディレクトリ構成
- 詳細情報の場所
- 環境変数と秘密情報
- Git 運用
- プロジェクト固有の注意点
- 禁止事項

## 品質基準

- 読み手が AI コーディングエージェントであることが明確である。
- 1 つの箇条書きが 1 つの行動に対応している。
- コマンド、ツール、ファイル名が現在のリポジトリと一致している。
- 秘密情報や token の値を含まない。
- 将来すぐ古くなるメモを大量に含まない。
- 詳細を本文に抱え込まず、必要な文書やファイルへの参照を置いている。
- 自動生成したままにせず、各行が本当に必要か確認されている。
